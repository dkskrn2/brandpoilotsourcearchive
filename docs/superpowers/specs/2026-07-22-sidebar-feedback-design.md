# 사이드바 피드백 설계

## 목표

고객이 현재 화면을 떠나지 않고 사이드바에서 피드백을 전송하고 통화 문의를 예약할 수 있게 한다. 피드백은 고객센터 문의와 별도 데이터로 저장하며 관리자 API에서도 독립 목록으로 제공한다.

## 고객 화면

- 데스크톱·모바일 사이드바 하단 유틸리티 영역에 `피드백` 버튼을 둔다.
- 접힌 데스크톱 사이드바에서는 아이콘과 접근성 이름만 유지한다.
- 버튼을 누르면 현재 화면 위에 접근 가능한 모달을 연다.
- 모달 제목은 `피드백`, 안내문은 `여러분의 의견이 모종을 더 좋게 만듭니다.`로 표시한다.
- 첫 카드 제목은 `통화 문의 예약하기`이며 15분 통화 안내와 예약 버튼을 제공한다.
- 예약 URL은 `VITE_FEEDBACK_BOOKING_URL`로 설정한다. 값이 없으면 고객센터 문의 작성 영역으로 이동해 끊어진 버튼이 생기지 않게 한다.
- 두 번째 카드에는 최대 2,000자의 의견 입력란과 `보내기` 버튼을 둔다.
- 공백 입력은 전송하지 않으며, 전송 중 중복 제출을 막고 성공·실패 상태를 모달 안에서 알린다.
- 성공 후 입력값을 비우고 성공 메시지를 표시한다.
- 닫기 버튼, Escape, 배경 클릭으로 닫을 수 있고 열릴 때 입력란으로 포커스가 이동하며 닫힌 뒤 피드백 버튼으로 돌아간다.

## 데이터와 API

`feedback_submissions`를 `support_requests`와 분리한다. 필드는 `id`, `workspace_id`, `brand_id`, `message`, `status`, `created_at`, `updated_at`, `deleted_at`으로 제한한다. 상태는 `new`, `reviewed`, `archived`다.

- 고객 API: `POST /brands/:brandId/feedback`
  - `message`를 trim하고 1~2,000자만 허용한다.
  - 현재 브랜드의 workspace를 서버에서 결정한다.
- 관리자 API: `GET /admin/v1/feedback`
  - 기존 관리자 인증과 커서 페이지네이션을 사용한다.
  - `q`, `status`, `brandId`, `cursor`, `limit` 필터를 지원한다.
- 고객센터 문의 API와 테이블은 변경하지 않는다. 따라서 관리자 소비자는 `/admin/v1/feedback`과 기존 고객센터 문의 흐름을 서로 다른 메뉴로 표현할 수 있다.

현재 저장소에는 관리자 웹 UI가 없으므로 이번 범위는 관리자 피드백 전용 API까지다. 관리자 UI가 추가될 때 고객센터 문의 메뉴와 피드백 메뉴가 각각 해당 API를 사용한다.

## 오류 처리와 검증

- 빈 값과 2,000자 초과는 고객·서버 양쪽에서 거절한다.
- 존재하지 않는 브랜드는 `brand_not_found`로 처리한다.
- 고객 전송, 저장소 SQL, 관리자 목록과 필터, 모달 열기·닫기·예약·성공·실패를 자동 테스트한다.
- 마지막에는 고객 UI 전체 테스트, API 관련 테스트, 두 패키지 빌드, 실제 브라우저 전송을 확인한다.
