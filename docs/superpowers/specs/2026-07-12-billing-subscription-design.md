# Brand Pilot 단일 월 구독 결제 설계

작성일: 2026-07-12  
상태: 구현 승인 대기  
범위: 고객용 결제 및 구독 메뉴, 단일 월 자동결제, 이용 권한 판정

## 결정 사항

- 단일 월 구독 플랜만 제공한다.
- 무료 체험은 제공하지 않는다.
- 결제수단을 등록한 뒤 첫 달 결제가 승인되어야 서비스를 사용할 수 있다.
- 다음 달부터는 토스페이먼츠 자동결제로 청구한다.
- 정기 청구가 실패하면 즉시 서비스 이용 권한을 중지한다. 자동 재시도는 하지 않는다.
- 고객이 결제수단을 변경한 뒤 직접 재결제하면 이용 권한을 복구한다.
- 고객 해지는 다음 결제일부터 반영한다. 현재 결제 기간에는 계속 이용한다.
- 향후 관리자는 유료 구독과 별개로 무료 이용 기간을 줄 수 있다.

## 고객 IA

사이드바에 `결제 및 구독`을 `브랜드 설정` 아래, `고객센터` 위에 둔다. 결제 관련 고객 기능은 `/billing` 단일 화면으로 제공한다.

화면은 다음 정보를 순서대로 보여준다.

1. 이용 상태: `미구독`, `활성`, `해지 예정`, `결제 실패로 이용 중지`, `관리자 제공 이용 기간`
2. 단일 플랜: 플랜명, 월 가격, 다음 결제일 또는 이용 종료일
3. 결제수단: 등록 여부, 카드사와 끝 4자리, 등록 또는 변경 버튼
4. 구독 관리: 구독 시작, 해지 예약, 해지 취소
5. 결제 실패 복구: 실패 사유와 결제수단 변경·재결제 버튼
6. 결제 이력: 결제일, 주문번호, 금액, 상태, 영수증 링크

고객은 가격, 청구일, 결제수단 원문, 빌링키를 직접 수정하거나 볼 수 없다.

## 상태와 권한

`subscriptions.status`는 `pending_payment`, `active`, `cancel_scheduled`, `suspended`, `cancelled` 중 하나다.

`workspace_entitlements`는 실제 제품 접근 여부를 결정한다. 구독이 `active` 또는 `cancel_scheduled`이고 현재 기간이 끝나지 않았으면 `subscription` 근거의 활성 권한을 둔다. 관리자가 무료 기간을 부여하면 `admin_grant` 근거의 별도 활성 권한을 둔다. API는 구독 상태만 보지 않고 활성 권한이 하나 이상 있는지 검사한다.

```text
미구독 -> 결제수단 등록 -> 첫 결제 승인 -> 활성
활성 -> 정기 청구 성공 -> 활성 유지
활성 -> 정기 청구 실패 -> 이용 중지
이용 중지 -> 고객 재결제 승인 -> 활성
활성 -> 해지 요청 -> 해지 예정 -> 기간 종료 -> 해지 완료
```

## 결제 흐름

1. 고객이 서버에 구독 시작 주문 생성을 요청한다.
2. 서버는 DB의 활성 단일 플랜 가격으로 주문을 저장한다. 프론트 금액을 받지 않는다.
3. 프론트는 토스 SDK로 빌링키 등록을 요청한다.
4. 성공 콜백에서 서버는 토스 응답, 주문번호, 금액을 DB 주문과 대조한다.
5. 서버가 첫 결제를 승인하고, 암호화한 빌링키를 저장한 뒤 구독과 이용 권한을 활성화한다.
6. Cron이 결제일이 된 활성 구독을 한 번만 청구한다.
7. 청구 성공은 기간과 이용 권한을 갱신한다. 실패는 즉시 `suspended`로 바꾸고 구독 기반 이용 권한을 만료한다.
8. 토스 웹훅은 이벤트를 기록한 뒤 `paymentKey`로 토스 조회 API를 다시 호출해 상태를 보정한다.

결제 성공 URL의 `paymentKey`, `orderId`, `amount`는 신뢰할 수 있는 값이 아니다. 서버 주문의 가격 및 주문번호와 일치할 때만 결제를 승인한다. 결제·취소·웹훅 처리에는 주문별 멱등키를 둔다.

## 데이터 모델

| 테이블 | 핵심 책임 |
|---|---|
| `plans` | 단일 월 플랜 코드, 이름, KRW 금액, 활성 여부 |
| `subscriptions` | 워크스페이스별 상태, 현재 이용 기간, 다음 청구일, 해지 예약 |
| `billing_customers` | 토스 `customerKey`, 암호화된 빌링키, 결제수단 마스킹 정보 |
| `payment_orders` | 서버가 고정한 금액과 주문 상태 |
| `payments` | 토스 `paymentKey`, 승인·실패·취소 결과 |
| `payment_webhook_events` | 웹훅 원문, 중복 방지 키, 처리 결과 |
| `workspace_entitlements` | 구독 또는 관리자 제공 기간에 따른 실제 제품 접근 권한 |

모든 테이블은 `workspace_id`를 포함하고, 비밀값인 토스 시크릿 키와 빌링키는 브라우저·로그·평문 DB에 저장하지 않는다.

## API 경계

- `GET /brands/:brandId/billing/summary`: 고객용 구독·권한·결제수단 마스킹 정보·최근 결제 이력
- `POST /brands/:brandId/billing/subscription/start`: 서버 가격으로 첫 결제 주문 생성
- `POST /brands/:brandId/billing/payment-method/confirm`: 빌링키 등록 확인 및 첫 결제 승인
- `POST /brands/:brandId/billing/subscription/cancel`: 다음 결제일부터 해지 예약
- `POST /brands/:brandId/billing/subscription/resume`: 기간 종료 전 해지 예약 취소
- `POST /brands/:brandId/billing/payment/retry`: 결제수단 변경 뒤 고객이 요청한 단일 재결제
- `POST /webhooks/toss`: 토스 이벤트 기록 및 결제 상태 재검증
- `POST /internal/cron/billing`: 결제일 도래 구독 청구

## 운영 의존성

결제 메뉴와 DB는 가격 없이도 `결제 준비 중` 상태로 표시할 수 있다. `BILLING_ENFORCEMENT_ENABLED=false`인 동안에는 기존 고객 기능을 차단하지 않는다. 실제 첫 결제와 자동 청구를 활성화하고 이 설정을 `true`로 전환하려면 다음이 필요하다.

- 월 가격과 VAT 포함 여부
- 토스페이먼츠 자동결제 계약 MID
- 테스트 및 라이브 클라이언트 키·시크릿 키
- 운영 도메인의 성공 URL, 실패 URL, HTTPS 웹훅 URL
- 환불 및 개인정보 처리방침·이용약관

별도 Docker 컨테이너는 만들지 않는다. 기존 API가 토스 서버 API와 웹훅을 처리하고, 기존 PostgreSQL이 결제 이력을 저장한다. 청구 Cron은 배포 환경 스케줄러가 API 내부 엔드포인트를 호출한다.

## 범위 분리

첫 구현은 메뉴, 권한, DB, 구독 상태 API, 테스트 키 기반 첫 결제 흐름까지다. 관리자 무료 기간 부여 UI, 환불 운영 UI, 다중 플랜, 세금계산서, 자동 재시도는 후속 범위다.
