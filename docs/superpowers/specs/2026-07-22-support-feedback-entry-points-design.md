# 고객센터와 대시보드 피드백 진입점 설계

## 목표

대시보드와 고객센터에서 `기능 제안하기`를 누르면 화면을 이동하지 않고 기존 사이드바의 피드백 팝업을 연다. 일반 고객 문의와 기능 피드백의 입력 경로를 분리하고, 고객센터의 불필요한 수동 새로고침 조작을 제거한다.

## 화면과 동작

- 대시보드 하단의 `기능 제안하기` 링크를 버튼으로 변경한다.
- 고객센터 문의 유형에서 `기능 건의`를 제거한다.
- 고객센터의 문의 내역 `새로고침` 버튼을 제거한다.
- 고객센터 하단에 대시보드와 같은 문구와 형태의 기능 제안 배너를 추가한다.
- 대시보드와 고객센터의 `기능 제안하기` 버튼은 모두 사이드바 `피드백` 버튼과 동일한 `FeedbackDialog`를 연다.
- 피드백 팝업을 닫으면 사용자가 눌렀던 버튼으로 키보드 포커스를 복원한다.

고객센터 문의 내역은 화면 진입 시 계속 자동으로 불러온다. 문의가 등록된 뒤에도 목록을 자동으로 다시 불러오므로 수동 새로고침 버튼 제거가 데이터 갱신 동작을 없애지는 않는다.

## 컴포넌트와 상태

`AppShell`에서 피드백 팝업 한 개만 렌더링한다. 피드백 열기 함수를 전용 React 컨텍스트로 제공하며 `Sidebar`, `DashboardPage`, `SupportPage`가 같은 함수를 호출한다. 각 페이지에 팝업 상태나 제출 로직을 복제하지 않는다.

기존 `FeedbackDialog`, 피드백 제출 API, 통화 예약 링크와 성공·실패 처리는 변경하지 않는다. `SupportRequestCategory`와 서버의 기존 `feature` 값도 과거 문의 호환성을 위해 유지하고, 고객센터의 새 문의 선택지에서만 숨긴다.

## 접근성과 반응형 동작

- 링크에서 버튼으로 바뀐 대시보드 진입점은 기존 버튼 스타일과 접근성 이름을 유지한다.
- 고객센터 배너는 기존 대시보드 배너의 반응형 스타일을 재사용한다.
- 팝업의 제목, Escape 닫기, 배경 클릭 닫기와 포커스 이동 동작은 기존 구현을 유지한다.
- 데스크톱과 모바일에서 버튼 문구와 배너 안내가 겹치지 않아야 한다.

## 검증 기준

- 고객센터 문의 유형에 `기능 건의`가 표시되지 않는다.
- 고객센터 문의 내역에 `새로고침` 버튼이 표시되지 않는다.
- 고객센터는 진입 시 문의 내역을 자동 조회하고, 문의 등록 후 목록을 자동 갱신한다.
- 대시보드와 고객센터의 `기능 제안하기` 버튼이 페이지 이동 없이 같은 피드백 팝업을 연다.
- 사이드바의 기존 피드백 버튼도 계속 같은 팝업을 연다.
- 고객 UI 테스트와 빌드가 통과한다.

## 제외 범위

- 기존 피드백 저장 API와 데이터베이스 구조 변경
- 과거 `feature` 유형 문의 데이터의 변환 또는 삭제
- 고객센터 문의 내역의 자동 주기 갱신
- 피드백 팝업 디자인과 입력 항목의 재설계
