# 브랜드 정보 자동 분석 온보딩 설계

## 목적

신규 사용자가 회사 URL 또는 문서를 입력하면 CLI 워커가 기업 정보를 구조화하고, 사용자가 결과를 확인·수정한 뒤 확정된 정보를 서비스 전체에서 재사용한다.

이 기능은 현재 분리되어 있는 자사 URL 크롤링, 브랜드 Wiki 생성, AI 콘텐츠의 자사 정보 조회를 하나의 공통 브랜드 정보 흐름으로 통합한다. 특정 제품·서비스 URL 분석은 콘텐츠마다 목적이 다르므로 기존 별도 흐름을 유지한다.

## 확정 범위

- 전용 3단계 화면: `자료 입력 → CLI 분석 → 확인·저장`
- 입력 자료
  - 자사 URL 1개
  - 문서 최대 5개
  - 지원 형식: TXT, MD, 텍스트형 PDF, CSV, XLSX
  - 스캔 PDF OCR, DOCX, PPTX는 1차 범위에서 제외
- CLI 분석 결과
  - 기업 개요
  - 사업 소개
  - 대표 분야와 세부 분야
  - 핵심 타깃
  - 차별점
  - 핵심 소구점
  - 경쟁사와 대안
  - 근거 URL과 정보 부족 항목
- 경쟁사와 시장 정보는 공개 웹검색을 허용하되 근거 URL을 필수로 기록한다.
- 핵심 타깃, 차별점, 핵심 소구점, 기업 개요, 사업 소개, 경쟁사 설명은 사용자가 textarea에서 수정할 수 있다.
- 사용자가 `확인하고 저장`하기 전에는 현재 브랜드 정보를 변경하지 않는다.
- 일반 URL 크롤링과 Wiki 갱신은 기존 72시간 주기를 유지한다.
- CLI 기업 분석은 최초 온보딩, 입력 자료 변경 후 명시적 분석, 사용자의 `다시 분석` 요청에서만 실행한다.

## 사용자 흐름

### 1. 자료 입력

온보딩 체크리스트의 `브랜드 정보 만들기`에서 전용 화면으로 이동한다.

- URL과 문서는 함께 사용할 수 있다.
- URL은 현재 브랜드의 자사 URL 한 개와 동일한 자원으로 관리한다.
- 파일별 형식, 용량, 개수를 입력 시점에 검증한다.
- URL이나 파일이 하나 이상 있어야 분석을 시작할 수 있다.
- 이미 확정된 브랜드 정보가 있으면 이 화면은 `다시 분석` 흐름으로 열린다.

### 2. CLI 분석

화면은 작업 상태만 조회하며 장시간 HTTP 요청을 유지하지 않는다.

1. API가 분석 실행과 입력 자료 스냅샷을 생성한다.
2. URL 크롤러와 문서 파서가 자료를 공통 텍스트·표 계약으로 정규화한다.
3. 브랜드 분석 CLI 워커가 작업을 임대한다.
4. 워커가 입력 근거를 분석하고 필요한 경쟁사·시장 정보만 공개 웹검색한다.
5. 워커가 구조화 결과와 근거를 제출한다.
6. 프론트는 상태를 폴링하고 완료 시 확인 화면으로 이동한다.

분석 실패 시 기존 브랜드 프로필은 유지한다. 실패 코드와 재시도 가능 여부를 실행 기록에 저장하고 사용자는 같은 자료로 다시 요청할 수 있다.

### 3. 확인·저장

결과 화면은 다음 섹션을 제공한다.

- 기업·사업 개요
- 분야
- 핵심 타깃
- 차별점
- 핵심 소구점
- 경쟁사·대안
- 근거와 누락 정보

각 분석 필드는 편집 가능하다. 근거는 필드 아래의 보조 링크나 별도 근거 패널에서 확인한다. `다시 분석`은 새 초안을 만들 뿐 현재 확정 정보를 덮어쓰지 않는다.

`확인하고 저장`은 한 트랜잭션으로 다음을 수행한다.

1. 분석 실행을 확정 버전으로 표시한다.
2. 기존 `brand_profiles` 호환 필드에 기업 설명, 대표·세부 분야, 핵심 고객을 동기화한다.
3. 확정된 구조화 브랜드 정보를 활성 버전으로 지정한다.
4. 브랜드 Wiki 재생성 작업을 요청한다.
5. 온보딩의 브랜드 프로필 단계를 완료 처리한다.

## 아키텍처

### 공통 자료 추출 계층

기존 안전한 URL 크롤러의 URL 검증, DNS 고정, 리디렉션 제한, 응답 크기 제한을 재사용한다. AI 콘텐츠 제품 분석의 이미지 추출은 이 계층 위에서 제품 분석 전용 기능으로 유지한다.

문서 파서는 모든 입력을 다음 공통 구조로 변환한다.

```ts
interface BrandEvidenceDocument {
  sourceId: string;
  sourceType: "owned_url" | "text" | "markdown" | "pdf" | "csv" | "xlsx";
  title: string;
  sourceUrl: string | null;
  textBlocks: Array<{ heading: string | null; text: string }>;
  tables: Array<{ sheet: string | null; headers: string[]; rows: string[][] }>;
  contentHash: string;
}
```

원본 업로드 파일은 분석 중 임시 저장한다. 분석 완료 후 원본은 삭제하고, 정규화한 텍스트·표와 콘텐츠 해시만 브랜드 근거로 보관한다. 확정된 근거는 Wiki 생성에도 사용할 수 있다.

### 브랜드 분석 CLI 워커

제품 분석 워커를 확장하지 않고 별도 브랜드 분석 워커를 둔다.

- 제품 분석 계약은 제품 이미지, 제품별 타깃 3개, 소구점 선택에 초점이 있다.
- 브랜드 분석 계약은 회사 전체의 지속적으로 재사용할 기준 정보에 초점이 있다.
- 두 워커는 URL 안전 추출과 작업 임대·heartbeat·timeout 패턴을 공유한다.
- 이미지 생성 워커와는 별도 리소스 유형을 사용하므로 이미지 생성 큐를 점유하지 않는다.

입력 계약에는 브랜드 기본 정보, 정규화 문서 목록, 공개 웹검색 정책이 포함된다. 출력 계약은 필드별 내용, 경쟁사 근거, source gaps를 엄격히 검증한다.

### 공통 브랜드 정보 제공자

서비스의 각 기능이 DB 테이블을 직접 조합하지 않도록 `BrandIntelligenceProvider` 경계를 둔다.

- 온보딩과 브랜드 설정: 확정 정보 조회·수정
- AI 콘텐츠 자사 정보: 확정 스냅샷을 즉시 조회하며 CLI 재분석하지 않음
- 자동 콘텐츠 생성: 확정 스냅샷 사용
- DM Wiki: 확정 정보와 근거 문서 포함
- 분야·타깃·소구점 추천: 확정 구조화 필드 사용

특정 제품·서비스 URL을 입력한 AI 콘텐츠 생성은 기존 제품 분석 흐름을 호출하고, 브랜드 정보는 배경 맥락으로만 추가한다.

## 데이터 모델

### `brand_analysis_runs`

- `id`, `workspace_id`, `brand_id`
- `status`: queued, extracting, analyzing, review_ready, confirmed, failed
- `input_manifest_json`
- `normalized_evidence_json`
- `result_json`: CLI 원본 결과
- `edited_result_json`: 사용자 수정 결과
- `skill_version`, `model`, `error_code`
- `created_at`, `started_at`, `finished_at`, `confirmed_at`

브랜드마다 확정된 최신 실행 한 건만 활성 버전이 된다. 이전 확정 버전은 감사와 비교를 위해 유지한다.

### `brand_analysis_uploads`

- 실행에 첨부된 임시 파일 메타데이터
- 원본 저장 경로, 파일명, MIME, 크기, 해시, 파싱 상태
- 분석 완료 또는 실패 정리 작업 후 원본 저장 경로를 비운다.

### 기존 `brand_profiles`

기존 소비자를 깨지 않도록 유지한다. 확정 시 다음 호환 필드를 동기화한다.

- `description` ← 기업·사업 개요 요약
- `primary_customer` ← 핵심 타깃
- 대표·세부 분야 관계

차별점, 소구점, 경쟁사 등 확장 정보는 확정된 `edited_result_json`에서 제공한다. `brand_profiles`에 동일 데이터를 중복 열로 추가하지 않는다.

## API 경계

- `POST /brands/:brandId/brand-analysis/uploads`
  - 문서 임시 업로드
- `POST /brands/:brandId/brand-analyses`
  - URL과 업로드 ID로 분석 실행 생성
- `GET /brands/:brandId/brand-analyses/:analysisId`
  - 상태, 분석 결과, 오류 조회
- `PATCH /brands/:brandId/brand-analyses/:analysisId/draft`
  - 사용자 수정 초안 저장
- `POST /brands/:brandId/brand-analyses/:analysisId/confirm`
  - 최종 확인 및 공통 브랜드 정보 활성화
- `GET /brands/:brandId/brand-intelligence`
  - 현재 확정된 공통 브랜드 정보 조회
- `POST /brands/:brandId/brand-analyses/:analysisId/retry`
  - 동일 입력 스냅샷으로 새 실행 생성

확정과 재시도 API는 idempotency key를 받아 중복 실행과 중복 Wiki 요청을 방지한다.

## 브랜드 설정 통합

브랜드 설정 상단의 브랜드 프로필 영역에 확정된 분석 정보를 표시한다.

- 기업 개요와 사업 소개
- 대표·세부 분야
- 핵심 타깃
- 차별점
- 핵심 소구점
- 경쟁사·대안
- 마지막 분석 일시와 사용 자료

기존 브랜드명, 로고, 색상, 톤, CTA, 자동 승인과 채널 형식 설정은 그대로 유지한다. `브랜드 정보 다시 분석` 버튼은 동일한 3단계 화면을 연다.

## AI 콘텐츠 통합

- `자사 정보 사용`은 `GET /brand-intelligence`만 호출한다.
- 저장된 확정 정보가 없으면 생성 단계에서 분석하지 않고 온보딩 분석 화면으로 안내한다.
- 제품·서비스 URL은 기존 제품 분석 캐시와 CLI 워커를 사용한다.
- 콘텐츠 생성 입력에는 확정 브랜드 정보의 버전 ID와 스냅샷을 기록해 나중에 어떤 기준으로 생성했는지 확인할 수 있게 한다.

## 오류와 안전 처리

- URL은 현재 크롤러의 SSRF, DNS 재바인딩, 리디렉션, 크기 제한을 적용한다.
- 파일 확장자만 믿지 않고 MIME과 실제 파싱 성공 여부를 검증한다.
- CSV/XLSX는 셀·행·시트 수와 전체 문자 예산을 제한한다.
- PDF는 텍스트가 거의 없으면 `scanned_pdf_not_supported`로 안내한다.
- CLI 출력은 허용 필드와 길이, URL 형식, 근거 필수 조건을 검증한다.
- 공개 웹검색 결과는 사용자 입력 자료와 구분해 표시한다.
- 분석 실패나 재분석 중에도 현재 확정 버전을 계속 제공한다.
- 사용자 수정값은 재분석 결과가 자동으로 덮어쓰지 않는다.

## 검증 범위

### API·DB

- URL만, 문서만, URL+문서 조합 실행
- 파일 개수·형식·크기 제한
- 분석 작업 임대, heartbeat, timeout, 재시도
- CLI 계약 오류 처리
- 수정 초안 저장과 확정 트랜잭션
- 중복 확정 요청 idempotency
- 기존 브랜드 프로필 호환 필드 동기화
- Wiki 재생성 요청 1회 보장

### 프론트

- 3단계 이동과 새로고침 후 상태 복원
- 장시간 분석 상태 폴링
- 모든 분석 필드 수정
- 근거 표시와 누락 정보 안내
- 저장 전 기존 프로필 미변경
- 확정 후 온보딩 완료 및 브랜드 설정 반영
- 모바일에서 단일 열 편집 화면

### 통합

- 확정 브랜드 정보가 AI 콘텐츠 자사 정보에 즉시 나타남
- 자동 콘텐츠와 DM Wiki가 동일 버전을 사용함
- 제품 URL 분석은 기존 별도 흐름을 유지함
- 일반 72시간 크롤링이 브랜드 CLI 분석을 자동 실행하지 않음

## 제외 범위

- 스캔 PDF OCR
- DOCX, PPTX
- 주기적 CLI 기업 재분석
- 사용자 확인 없는 자동 프로필 덮어쓰기
- 특정 제품·서비스 URL 분석 통합
- 경쟁사 정보를 이용한 자동 비교 광고 생성
