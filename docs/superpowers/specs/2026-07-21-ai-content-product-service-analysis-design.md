# AI 콘텐츠 제품·서비스 분석 설계

## 목적

AI 콘텐츠 생성에서 제품과 무형 서비스를 동일한 기준으로 분석하는 문제를 해소한다. 확정된 자사 브랜드 정보는 모든 생성의 공통 배경으로 사용하고, 사용자가 입력한 제품·서비스 자료는 생성 건마다 유형에 맞는 CLI 프롬프트로 새로 분석한다.

사용자는 분석 결과를 별도로 검토하지 않는다. 자료 입력 후 한 번 실행하면 내부 분석과 타깃·소구점 생성이 순차적으로 완료되고, 사용자는 타깃과 소구점부터 확인한다.

## 범위

### 포함

- 제품과 서비스의 분석 입력 및 프롬프트 분리
- URL, 첨부 문서, 이미지, 직접 설명의 통합 분석
- 확정된 브랜드 정보의 공통 기본 컨텍스트 적용
- 무형 서비스 유형 자동 판별
- 제품·서비스별 타깃 및 소구점 생성
- 소구점 수정·추가·삭제·재생성
- 타깃·소구점 다음 단계로 레퍼런스 선택 연결
- 생성 건별 임시 첨부파일 수명 관리
- 기존 생성 초안과의 하위 호환

### 제외

- 분석 결과 확인·수정 화면
- 서비스 유형의 사용자 확인·수정 UI
- 추출 사실의 사용자 수정 UI
- 같은 URL을 이용한 생성 건 간 분석 결과 재사용
- 제품과 서비스 외의 별도 제3유형
- 편집기 제공

## 사용자 흐름

AI 콘텐츠 생성 단계는 다음 순서를 따른다.

1. 콘텐츠 유형 선택
2. 제품·서비스 자료 입력 및 분석 실행
3. 타깃·소구점 선택 및 수정
4. 레퍼런스 선택
5. 프롬프트·색상·추가 이미지 설정
6. 최종 확인 및 콘텐츠 생성

### 1. 콘텐츠 유형 선택

카드뉴스, 블로그, 마케팅 소재 중 생성할 결과물을 선택한다. 영상 생성은 이 흐름에 포함하지 않는다.

### 2. 제품·서비스 자료 입력

사용자는 분석 대상을 `제품` 또는 `서비스`로 선택한다. 확정된 자사 브랜드 정보는 두 유형 모두에 자동 적용하며, 이를 제품·서비스 구분 기준으로 사용하지 않는다.

공통 입력은 다음과 같다.

- 공개 URL 1개
- 이름
- 가격·프로모션 또는 이용 조건
- 추가 설명
- 이미지 첨부
- 문서 첨부

URL과 첨부파일 중 하나만 있어도 분석할 수 있다. 두 가지를 함께 제공하면 하나의 근거 묶음으로 합쳐 분석한다. URL과 첨부파일이 모두 없으면 이름과 추가 설명만으로도 실행할 수 있지만, 확인 가능한 근거가 부족하다는 사실을 내부 분석 결과에 기록한다.

지원 문서는 PDF, TXT, Markdown, CSV, XLSX로 제한한다. 이미지는 기존 AI 콘텐츠 첨부 규격을 재사용한다.

사용자가 `분석하고 소구점 만들기`를 누르면 다음 내부 작업을 연속 실행한다.

1. URL 및 첨부자료 추출
2. 제품 또는 서비스 전용 분석 CLI 실행
3. 분석 결과 정규화 및 저장
4. 제품 또는 서비스 전용 타깃·소구점 CLI 실행
5. 타깃·소구점 단계로 이동

사용자에게는 세부 분석 결과 대신 다음 진행 상태만 표시한다.

- 제품·서비스 자료 확인 중
- 고객과 시장 분석 중
- 타깃·소구점 생성 중

### 3. 타깃·소구점

분석 결과를 기반으로 추천 타깃 3개와 각 타깃에 맞는 소구점을 제공한다.

사용자는 다음 작업을 할 수 있다.

- 타깃 1개 선택
- 소구점 1개 선택
- 소구점 제목 수정
- 소구점 설명 수정
- 소구점 직접 추가
- 필요 없는 소구점 삭제
- 소구점 다시 생성

수정·추가·삭제는 CLI를 호출하지 않고 생성 초안에 바로 저장한다. `소구점 다시 생성`을 누른 경우에만 확정된 내부 분석 결과를 이용해 소구점 CLI를 다시 실행한다.

타깃 1개와 소구점 1개가 선택되어야 다음 단계로 이동할 수 있다.

### 4. 레퍼런스 선택

타깃·소구점 다음에는 반드시 레퍼런스 선택 단계가 온다. 기존에 구현된 업종별, 컨셉별, 직접 찾기, 내 보관함과 직접 업로드 기능을 유지한다.

### 5. 프롬프트 설정

선택한 타깃, 수정된 소구점, 레퍼런스, 브랜드 대표 색상, 사용자가 추가한 제품·인물·참고 이미지를 최종 생성 워커 입력으로 조합한다.

## 브랜드 컨텍스트

확정된 브랜드 정보는 모든 분석에 공통으로 전달한다.

- 기업 개요
- 사업 소개
- 대표 분야와 세부 분야
- 핵심 타깃
- 차별점
- 기존 핵심 소구점
- 브랜드 색상
- 브랜드 정보 버전

브랜드 정보는 제품 또는 서비스 자료를 대체하지 않는다. 제품·서비스 분석의 배경과 브랜드 일관성을 제공하는 역할만 한다.

## 자료 우선순위

서로 다른 자료의 내용이 충돌하면 다음 우선순위를 적용한다.

1. 사용자 직접 입력
2. 첨부 문서
3. 사용자가 입력한 제품·서비스 URL
4. 확정된 브랜드 정보
5. 공개 웹 조사

공개 웹 조사는 VOC, 대안, 경쟁 맥락, 시장 언어에만 사용한다. 가격, 효능, 성과, 계약 조건을 공개 검색 결과만으로 확정하지 않는다.

## 제품 분석

제품 분석은 `기능 → 효익 → 구매 이유`를 중심으로 한다.

### 추출 항목

- 제품명과 카테고리
- 규격, 구성, 소재, 옵션
- 가격, 할인, 배송, 환불, 프로모션
- 기능과 기능별 효익
- 사용 방법과 사용 상황
- 구매 장벽과 반론
- 경쟁 제품과 대체재
- 제품 후기의 반복 만족·불만
- 제품 및 상세 이미지 후보

### 타깃 기준

- 사용 목적
- 생활 상황
- 구매 계기
- 불편과 욕구
- 가격 민감도
- 구매를 망설이는 이유

### 소구점 구성

`구체적인 고객 상황 → 제품 기능 → 고객이 얻는 변화 → 확인 가능한 근거` 순서로 작성한다.

## 서비스 분석

서비스 분석은 `문제 → 제공 과정 → 이용 후 변화 → 신뢰와 도입 부담 해소`를 중심으로 한다.

서비스는 SaaS뿐 아니라 컨설팅, 교육, 대행, 구독, 전문 서비스를 포함한다.

### 자동 판별 유형

- `saas`: SaaS·소프트웨어
- `consulting`: 컨설팅
- `education`: 교육·코칭
- `agency`: 대행·외주
- `subscription`: 정기 구독
- `professional`: 법무·세무 등 전문 서비스
- `other_service`: 기타 무형 서비스

서비스 유형은 내부 분석용이며 사용자에게 별도 확인 화면을 제공하지 않는다.

### 공통 추출 항목

- 해결하는 고객 문제
- 현재 고객이 사용하는 기존 방식
- 서비스 제공 과정
- 제공 결과물
- 사용자와 구매 결정권자
- 도입 전후 업무 흐름
- 이용·도입 조건
- 가격, 계약, 갱신, 해지 조건
- 전문성, 보안, 지원, 성과 등 신뢰 근거
- 서비스 도입을 망설이는 이유
- 경쟁 서비스와 수작업·외주 등의 대안

### 유형별 강조 기준

| 서비스 유형 | 강조 기준 |
| --- | --- |
| SaaS | 기능, 업무 흐름, 연동, 권한, 온보딩, 도입 난이도 |
| 컨설팅 | 진단 방법, 전문성, 수행 과정, 맞춤성, 산출물 |
| 교육 | 커리큘럼, 대상 수준, 학습 결과, 강사 신뢰도 |
| 대행 | 업무 범위, 산출물, 대응 방식, 관리 부담 감소 |
| 구독 | 반복 제공 가치, 갱신 이유, 해지 조건, 이용 빈도 |
| 전문 서비스 | 자격·경력, 정확성, 위험 감소, 상담 절차 |

### 타깃 기준

- 담당자의 역할
- 회사 또는 조직 규모
- 업무 성숙도
- 현재 사용하는 도구와 방식
- 병목이 발생하는 단계
- 실제 사용자와 구매 결정권자
- 도입을 망설이는 이유

### 소구점 구성

`현재의 병목 → 기존 방식의 한계 → 서비스 제공 과정 → 운영상 변화 → 신뢰 근거와 부담 해소` 순서로 작성한다.

## CLI 프롬프트 구조

프롬프트는 네 개로 분리한다.

- `product-analysis.v2-ko`
- `service-analysis.v2-ko`
- `product-appeal.v2-ko`
- `service-appeal.v2-ko`

제품과 서비스의 탐색 기준과 소구점 전개가 섞이지 않도록 분석 프롬프트와 소구점 프롬프트를 모두 분리한다.

### 분석 프롬프트 출력

공통 영역은 다음을 포함한다.

- 분석 요약
- 확인된 사실과 출처
- 공개 VOC와 대안
- 고객 문제와 구매·도입 장벽
- 부족한 정보

유형별 영역에는 제품 프로필 또는 서비스 프로필을 저장한다. 서비스 결과에는 자동 판별된 서비스 유형과 유형별 분석 내용이 포함된다.

### 소구점 프롬프트 출력

- 추천 타깃 정확히 3개
- 타깃별 최소 2개 소구점
- 각 소구점의 근거
- 타깃과 소구점의 연결 이유
- 추정 또는 확인되지 않은 주장 표시

## 데이터 계약

기존 `subject-analysis.v1`은 읽기 호환을 유지하고 새 작업은 `subject-analysis.v2`를 사용한다.

입력 계약의 핵심 구조는 다음과 같다.

```json
{
  "contractVersion": "subject-analysis.v2",
  "brandContext": {},
  "subject": {
    "type": "product",
    "sourceUrl": "https://example.com/product",
    "attachmentIds": [],
    "manualInput": {
      "name": "",
      "promotionOrTerms": "",
      "description": ""
    }
  }
}
```

분석 결과와 소구점 결과는 분리해 저장한다. 생성 초안에는 분석 ID, 분석 버전, 선택한 타깃, 사용자가 수정한 최종 소구점을 저장한다.

## 작업과 저장 정책

- 생성 건마다 새로운 분석 작업을 만든다.
- 같은 URL을 사용하더라도 다른 생성 건의 분석 결과를 자동 재사용하지 않는다.
- 같은 생성 건을 다시 열 때는 해당 생성 건의 분석 결과를 복원한다.
- 중복 클릭 방지를 위한 멱등성 키는 유지한다.
- 워커 lease, heartbeat, timeout, 실패 재시도 정책을 기존 작업 구조와 동일하게 적용한다.
- URL 추출 하나가 실패해도 첨부자료나 직접 입력이 있으면 `partial` 분석으로 진행한다.
- 모든 근거가 없으면 소구점 생성을 중단하고 입력 보완을 안내한다.

## 첨부파일 수명

- 업로드 원본은 생성 건 전용 임시 저장소에 둔다.
- 분석과 최종 콘텐츠 생성 동안만 접근한다.
- 생성 완료 또는 생성 건 삭제 후 원본 파일을 제거한다.
- 추출한 텍스트, 확인된 사실, 출처 메타데이터는 생성 기록에 유지한다.
- 이미지가 최종 산출물에 실제 사용된 경우 결과물 자산은 삭제하지 않는다.

## 오류 처리

- URL 실패: 첨부 또는 직접 입력이 있으면 부분 분석 진행
- 첨부 일부 실패: 성공한 자료만 사용하고 실패 파일 표시
- 분석 CLI 실패: 현재 단계 유지, 재시도 제공
- 소구점 CLI 실패: 분석은 유지하고 소구점 생성만 재시도
- 근거 부족: 추측하지 않고 입력 보완 안내
- 제품·서비스 유형 변경: 기존 분석과 타깃·소구점 선택 초기화
- URL·첨부·직접 입력 변경: 기존 분석과 타깃·소구점 선택 초기화

## 테스트 범위

### 프론트

- 제품과 서비스 입력 UI 분기
- URL 또는 첨부만으로 실행 가능
- 분석 결과 화면이 렌더링되지 않음
- 진행 상태가 순서대로 표시됨
- 완료 후 타깃·소구점 단계로 자동 이동
- 소구점 수정·추가·삭제 및 재생성
- 타깃·소구점 다음에 레퍼런스 단계가 표시됨

### API

- 브랜드 컨텍스트와 생성 건별 자료 결합
- 제품·서비스별 작업 계약 생성
- 첨부 추출 결과 포함
- 작업 멱등성 및 같은 생성 건 복구
- 교차 생성 분석 캐시 미사용
- 분석 성공 후 소구점 작업 연쇄 생성
- 부분 실패와 재시도

### 워커

- 제품·서비스 분석 프롬프트 분리
- 서비스 유형 자동 판별
- 제품·서비스 소구점 프롬프트 분리
- 근거 없는 가격·효능·성과 생성 차단
- 타깃 3개와 타깃별 소구점 계약 검증

### E2E

- 제품 URL만 사용한 카드뉴스 생성
- 제품 문서와 이미지 첨부를 사용한 마케팅 소재 생성
- SaaS URL과 소개 문서를 사용한 블로그 생성
- 컨설팅 직접 설명과 PDF를 사용한 카드뉴스 생성
- 분석 실패 후 재시도
- 소구점 수정 후 레퍼런스 선택과 최종 생성까지 완료

## 완료 기준

- 제품과 서비스가 서로 다른 분석·소구점 프롬프트를 사용한다.
- 모든 분석에 확정된 브랜드 정보가 기본 컨텍스트로 포함된다.
- 서비스가 SaaS, 컨설팅, 교육, 대행, 구독, 전문 서비스를 자동 판별한다.
- 사용자는 분석 결과 화면을 거치지 않고 타깃·소구점부터 확인한다.
- 소구점을 수정한 뒤 레퍼런스를 선택하고 최종 콘텐츠를 생성할 수 있다.
- 생성 건 간 분석 결과가 의도치 않게 재사용되지 않는다.
- 기존 `subject-analysis.v1` 생성 기록을 계속 열 수 있다.
