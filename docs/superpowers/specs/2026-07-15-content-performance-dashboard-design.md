# 콘텐츠 성과 대시보드 설계

작성일: 2026-07-15
상태: 확정
대상: Brand Pilot 고객용 웹앱

## 1. 목적

고객이 최근 콘텐츠 운영 상태와 발행 성과를 한 화면에서 확인할 수 있는 첫 화면을 제공한다.

이번 범위는 다음 콘텐츠 추천이나 자동 전략 수립이 아니다. 최근 30일 동안 무엇이 발행됐고, 얼마나 조회 또는 노출됐으며, 지금 처리해야 할 검토·실패 항목이 무엇인지 보여주는 운영 대시보드다.

## 2. 확정 결정

| 항목 | 결정 |
|---|---|
| 화면 배치 | 운영 현황 중심 A안 |
| 기본 경로 | `/dashboard` |
| 로그인 후 첫 화면 | `/dashboard` |
| 집계 기간 | 최근 30일 단일 기준 |
| 성과 수집 | 매일 새벽 3시(KST) |
| 지원 채널 구조 | 현재 채널 카탈로그(Instagram, Threads, X, LinkedIn, YouTube, TikTok) 공통 구조 |
| 인증 전 채널 | `연결 전` 표시, 샘플 데이터 금지 |

| 다음 콘텐츠 제안 | 제외 |
| 트렌드·DM 데이터 활용 | 제외 |

## 3. 화면 정보 구조

사이드바 최상단에 `대시보드`를 추가한다.

대시보드는 아래 순서로 구성한다.

### 3.1 화면 헤더

- 제목: `전체 현황`
- 기간: `최근 30일 · YYYY년 M월 D일 기준`
- 마지막 성과 수집 시각

사용자가 기간을 직접 변경하는 기능은 이번 범위에 포함하지 않는다.

### 3.2 핵심 요약

- 발행 완료 수
- 조회·노출 합계
- 검토 필요 수
- 게시 실패 수

모든 수치는 최근 30일 기준이다. 전체 누적 수치를 함께 표시하지 않는다.

### 3.3 콘텐츠 운영 흐름

- 주제 대기
- 생성 중
- 검토 필요
- 예약·게시

각 값은 현재 상태를 보여주는 운영 지표다. 성과 지표의 30일 범위와 달리 현재 큐 상태를 집계하며, 화면에서 `현재`임을 명시한다.

### 3.4 조회·노출 추이

- 최근 30일 일별 조회·노출 증가량
- 채널별 시리즈
- 데이터가 있는 채널만 범례에 표시

### 3.5 채널별 성과

- 채널명
- 최근 30일 발행 수
- 최근 30일 조회·노출 수
- 연결 상태
- 마지막 정상 수집 시각

### 3.6 상위 콘텐츠

- 제목
- 채널과 콘텐츠 형식
- 발행일
- 조회·노출 수
- 원본 콘텐츠 또는 게시 상세 이동

최근 30일 안에 발행된 콘텐츠를 최신 조회·노출 누적값 기준으로 정렬한다.

### 3.7 확인 필요

- 게시 실패
- 채널 인증 또는 연결 오류
- 성과 수집 실패
- 오래된 마지막 수집 시각

## 4. 성과 지표 정규화

플랫폼별로 명칭과 제공 범위가 다른 지표를 무리하게 같은 의미로 취급하지 않는다.

대시보드의 공통 성과 값은 `exposure_count` 하나로 정규화한다.

- 플랫폼이 조회수를 제공하면 조회수를 사용한다.
- 조회수 대신 노출수를 제공하면 노출수를 사용한다.
- 어느 값도 제공하지 않으면 `NULL`로 저장하고 화면에 `데이터 없음`을 표시한다.
- 제공되지 않는 값을 `0`으로 저장하지 않는다.

채널 API가 반환하는 조회·노출 수는 콘텐츠별 누적값으로 취급한다. 따라서 날짜별 스냅샷의 `exposure_count`를 서로 더하지 않는다.

- 요약의 조회·노출 합계: 최근 30일 안에 발행된 콘텐츠별 최신 `exposure_count`의 합
- 상위 콘텐츠: 최근 30일 안에 발행된 콘텐츠별 최신 `exposure_count`
- 일별 추이: 같은 콘텐츠의 당일 누적값에서 직전 스냅샷 누적값을 뺀 증가량의 일별 합
- 첫 스냅샷: 이전 값이 없으므로 추이 증가량에서는 제외하고 기준점으로만 저장
- 플랫폼에서 누적값이 감소하면 음수 증가량을 만들지 않고 `0`으로 처리하며 원본값은 유지

좋아요, 댓글, 공유, 저장, 도달, 시청 시간은 플랫폼별 의미와 지원 범위가 다르므로 공통 대시보드에서 제외한다. 채널 API의 원본 지표가 향후 필요할 수 있으므로 `raw_metrics`에는 보관할 수 있지만, 이번 UI 집계에는 사용하지 않는다.

## 5. 데이터 모델

### 5.1 `content_performance_snapshots`

발행 콘텐츠의 날짜별 성과 스냅샷이다.

- `id`
- `brand_id`
- `channel_type`
- `publish_queue_id`
- `channel_output_id`
- `external_post_id`
- `snapshot_date`
- `exposure_count` nullable
- `raw_metrics` JSON
- `collected_at`
- `created_at`
- `updated_at`

중복 방지를 위해 `publish_queue_id + snapshot_date`에 유일 제약을 둔다. 같은 날 다시 수집한 경우 해당 날짜 스냅샷을 갱신한다.

`snapshot_date`는 KST 기준 날짜를 사용하고, `brand_id + channel_type + snapshot_date` 조회 인덱스를 둔다.

### 5.2 `performance_sync_runs`

채널별 성과 수집 실행 이력이다.

- `id`
- `brand_id`
- `channel_type`
- `status`: `queued`, `running`, `completed`, `partially_failed`, `failed`, `not_configured`
- `target_count`
- `success_count`
- `failure_count`
- `error_summary`
- `started_at`
- `completed_at`
- `created_at`

## 6. 수집 아키텍처

```text
로컬 중앙 API가 실행 중일 때 매일 03:00 KST
  -> API 내부 로컬 스케줄러가 연결된 브랜드·채널 조회
  -> 채널별 performance sync run 생성
  -> 최근 30일 발행 완료 콘텐츠 조회
  -> 채널 성과 어댑터 호출
  -> 콘텐츠별 날짜 스냅샷 upsert
  -> 실행 결과와 마지막 수집 시각 기록
```

채널별 어댑터는 공통 인터페이스를 사용한다.

```text
collectPerformance({ brandChannel, publishedItems })
  -> itemResults[]
     - publishQueueId
     - externalPostId
     - exposureCount | null
     - rawMetrics
     - collectedAt
```

- 한 채널 실패가 다른 채널 수집을 중단하지 않는다.
- 한 콘텐츠 실패가 같은 채널의 나머지 콘텐츠 수집을 중단하지 않는다.
- OAuth 또는 개발자 설정이 없는 채널은 외부 호출 없이 `not_configured`로 기록한다.
- 샘플, 임의 수치, 이전 하드코딩 값은 생성하지 않는다.
- Vercel Cron은 사용하지 않는다.
- 기존 `LOCAL_SCHEDULER_ENABLED=true` API 프로세스에 성과 수집 틱을 추가하며 별도 스케줄러 프로세스를 만들지 않는다.
- API가 03:00 KST에 꺼져 있었더라도 같은 KST 날짜에 성공 또는 부분 성공 실행 이력이 없으면 재시작 후 한 번 보충 실행한다.
- `brand_id + channel_type + KST 실행 날짜`를 실행 고유 키로 사용해 여러 틱과 서버 재시작이 같은 작업을 중복 생성하지 못하게 한다.

## 7. API

### 7.1 대시보드 조회

`GET /brands/:brandId/dashboard?period=30d`

응답:

- `period`
- `generatedAt`
- `lastCollectedAt`
- `summary`
  - `publishedCount`
  - `exposureCount`
  - `pendingReviewCount`
  - `failedPublishCount`
- `workflow`
- `dailyExposure`
- `channelPerformance`
- `topContents`
- `attentionItems`

`period`는 현재 `30d`만 허용한다.

## 8. 빈 상태와 오류 처리

### 전체 데이터 없음

- 최근 30일 발행 콘텐츠가 없음을 표시한다.
- 게시 관리로 이동할 수 있는 링크를 제공한다.

### 채널 연결 전

- 채널별 성과 행은 유지한다.
- 수치 대신 `연결 전`을 표시한다.
- 채널 연결 화면으로 이동할 수 있다.

### 성과 미지원 또는 값 없음

- `0`이 아닌 `데이터 없음`을 표시한다.

### 부분 실패

- 정상 수집된 채널 데이터는 계속 보여준다.
- 실패한 채널과 마지막 정상 수집 시각을 `확인 필요`에 표시한다.

### API 전체 실패

- 이전 데이터를 샘플로 대체하지 않는다.
- 오류 상태와 다시 시도 버튼을 표시한다.

## 9. 프론트엔드 변경

- `DashboardPage` 추가
- 사이드바 최상단에 대시보드 메뉴 추가
- 인증된 루트 경로와 로그인 성공 경로를 `/dashboard`로 변경
- API 클라이언트와 응답 타입 추가
- 차트는 기존 의존성을 우선 확인하고, 필요할 때만 가벼운 차트 라이브러리를 추가
- 기존 디자인 토큰과 8px 이하 카드 반경을 유지
- 모바일에서는 요약 지표를 2열, 나머지 영역을 1열로 재배치

## 10. 검증 범위

- DB 마이그레이션 및 유일 제약 테스트
- 최근 30일 경계와 집계 API 단위 테스트
- 누적값을 날짜별로 중복 합산하지 않는 테스트
- 전일 대비 증가량 계산 테스트
- 날짜별 스냅샷 upsert 중복 방지 테스트
- 미설정 채널이 외부 API를 호출하지 않는 테스트
- 빈 데이터와 부분 실패 UI 테스트
- React 및 API 빌드
- 데스크톱·모바일 핵심 화면 확인

실제 외부 플랫폼 성과 API 호출은 각 개발자 앱과 권한이 준비된 채널부터 별도로 연결한다. 이번 구현은 공통 구조, 실제 연결 가능한 채널 어댑터, 빈 상태 및 오류 처리까지 완료하는 것을 목표로 한다.

## 11. 제외 범위

- 다음 콘텐츠 자동 제안
- 트렌드 데이터 기반 추천
- DM 질문 기반 추천
- 기간 선택 UI
- 좋아요·댓글·저장·공유의 공통 집계
- 플랫폼별 상세 분석 화면
- 인증되지 않은 채널의 가상 성과 데이터
